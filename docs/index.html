Fast Retailing Co.(UNIQLO) Online Store

Back-End Service development

https://www.uniqlo.cn

2016.09 - 2018.04

Uniqlo official online store, developed based on Spring Cloud Microservice framework.
The system adopts hybrid cloud architecture. Using IDC computer room and Azure Cloud at the same time.

Official online store and WeChat Mini Programs are using the same Back-End services.
This project went live from 2018. At the time of Double 11 Shopping Festival in 2018, the order volume has already surpassed the Uniqlo Tmall store.

1. The system has two databases located in the Azure and IDC computer rooms respectively as hybrid cloud architecture. 混合云架构中，云端与机房均有数据库。弱一致性数据(购物车、收藏夹等)的操作请求会根据CDN定位进行就近分流、双端读写。

Design and implement a real-time data synchronization component based on Kafka. This component is aimd to solve data differences caused by double-end read and write between Azure and IDC databases. Ensuring data consistency through optimistic lock, tombstone and dead-letter queue. Eliminating the impact of redundant data at same time.

2. 由于项目中微服务数量较多(大促时500左右)，且未引入容器编排等组件。手工维护微服务实例部署、扩展、重启等操作成本过高。

为优化服务维护的复杂度，根据开源项目Spring-Boot-Admin(https://github.com/codecentric/spring-boot-admin)进行定制开发。集成Turbine组件实现微服务实例端点监控；集成Ansible调用shell脚本实现根据监控指标自动伸缩实例，手动⼀键部署等功能。

3. 项目初期使用Spring Cloud框架中的标准鉴权体系，Feign调用过程中转发AccessToken，每个微服务都与Auth服务交互进行token校验。导致并发量大的时候Auth服务负载过高，影响系统稳定性。

为改善高并发场景下Auth最先崩溃的情况，修改鉴权方式为gateway网关调用Auth服务验证AccessToken，并提取用户信息存入JWT。通过增加拦截器，实现Feign转发过程中将JWT重构到请求Header中，每个微服务仅静态校验JWT签名位。
